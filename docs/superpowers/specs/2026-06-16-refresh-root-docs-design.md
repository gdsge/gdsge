## Problem

describe code that has since been replaced or extended, so they misdescribe the current
toolbox. This is a documentation-sync task: bring the prose back in line with `src/` and the
`PROGRESS.md` changelog, with **no behavioral code changes**.

## Changes

### 1. `architecture.md` — three fact corrections

1. **Runtime module list:** remove `useFusedConstruct` (internal A/B switch) — it is deleted.
2. **Kernels paragraph:** rewrite the closing sentence about `myppual_mex`. State that the
   simulate / `output_interp` evaluation path uses the stacked-uniform-order
   `interp_construct_mex` (build) + `interp_eval_mex` (eval) on `interp_eval_double.h`, and
   that cartesian `SIMU_INTERP` runs the whole period loop in a generated
   `simulate_<model>_mex`. Remove the "prebuilt `myppual_mex` retained" claim.
3. **SymPy bridge:** change "in-process via pyenv" to **out-of-process** (`ExecutionMode =
   'OutOfProcess'`), noting it avoids the in-process native crash. Applies to the
   `gdsge.parser` line (~33), the backend-consumption bullet (~86), and the module map (~54).

### 2. `user-guide.md` — fact fix + new subsection

1. **ASG-under-SymPy:** correct the "spline only" statement to "spline and ASG interpolation;
   **pchip** still raises a clear error under `UseAutoDiff=0`", and trim the deferred-features
   sentence to pchip only.
2. **New "Choosing the C++ Jacobian backend" subsection**, accurate to `resolveBackend.m`:
   - Precedence: (1) in-gmod `UseAutoDiff` is authoritative (`1`⇒autodiff, `0`⇒SymPy);
     (2) `GDSGE_BACKEND` env (`adept`/`autodiff`/`sympy`/`auto`); (3) auto-detect.
   - Auto-detect: **Python present ⇒ SymPy** (with codegen-time fallback to adept on a SymPy
     failure); **Python absent ⇒ adept**.
   - Nuance to call out: with no pins set, installing Python flips the *default* to SymPy;
     auto mode falls back safely, whereas an explicit `UseAutoDiff` or `GDSGE_BACKEND` pin
     errors rather than falls back. The Python-free default path is still adept.
   - Update the `UseAutoDiff` options-table row to cross-reference this subsection.

### 3. `perf-report.md` — re-measure on current code

`tests/perf/run_perf`'s `writeReport` emits only the headline iter-time table plus one
"Reading" line; it does **not** reproduce the hand-written "Findings" narrative or the
profiling-breakdown table. A bare re-run therefore clobbers that analysis. Procedure:

1. Run `tests/perf/run_perf` (background; sequential ~11 child MATLAB processes; minutes) to
   produce authoritative fresh iter-time numbers. Success = result JSON exists/parses (a child
   may exit non-zero on MEX/OpenMP teardown after writing — known artifact).
2. Re-profile HL1996 / GLSW / Mendoza on current code (MATLAB `profile`) to refresh the
   profiling-breakdown percentages.
3. Re-author `perf-report.md` preserving its narrative structure: regenerated headline table
   + updated Findings + the profiling-breakdown table with the **footnote corrected** to the
   live kernels (`constructSplines` + `interp_construct_mex` + `interp_eval_mex`).
